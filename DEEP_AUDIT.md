# Deep Audit: Why LoRA Has Zero Effect

**Date:** 2026-03-09
**Context:** Phase 1 ablation (March 5) showed Forward Transfer = 0.0 across all conditions. All three conditions (Base, Base+RAG, Base+LoRA) resolved exactly 2/25 tasks.

---

## Executive Summary

LoRA produced zero improvement for three compounding reasons:

1. **Workspace contamination** — The RAG and LoRA conditions inherited dirty workspaces from the Base condition, making the cross-condition agreement (same 2 tasks) an artifact, not a meaningful result.
2. **The adapter actively damaged the model** — LoRA-adapted output was degenerate repetitive gibberish with zero tool calls. The model was worse, not better.
3. **Training data was garbage** — 86 shallow Q&A pairs synthesized by the same 7B model taught nothing useful. The synthesis pipeline produced "How do you read a file?" / "Use the read command" level content.

---

## Finding 1: Workspace Contamination Bug (CRITICAL)

### Evidence

The `create_workspace()` function in `looper/agent/workspace.py` (lines 41-50) is idempotent: if the workspace directory exists and HEAD matches `base_commit`, it returns immediately **without resetting uncommitted changes**. All conditions used the same `WORKSPACE_ROOT`.

This means:
- **Base** ran first, generated patches for 14/25 tasks (including django-12304 and django-13410)
- **Base+RAG** ran second on the same workspace — the Base condition's patches were still there
- **Base+LoRA** ran third on the same workspace — ditto

**Proof:** The generated patches in the RAG and LoRA trajectories for django-12304 and django-13410 are byte-for-byte identical to the Base condition's patches, despite:
- Base+RAG: 15 steps, 0 tool calls, outcome=`max_steps`
- Base+LoRA: 10 steps, 0 tool calls, all degenerate output, outcome=`max_steps`

Neither condition actually solved these tasks. They inherited the Base condition's work.

### Implication

The "2/25 across all conditions" result is meaningless as a comparison. We don't know if LoRA or RAG would have resolved any tasks independently, because they never had clean workspaces.

## Finding 2: The Adapter Produces Degenerate Output

### Adapter Weight Analysis

The Phase 1 adapter has 23,068,672 parameters across 224 tensors (16 layers x 14 tensors/layer):

| Metric | lora_a | lora_b |
|--------|--------|--------|
| Mean |abs| | ~0.009 | ~0.004 |
| Max |abs| | ~0.04 | ~0.03 |
| Std | ~0.011 | ~0.005 |
| Mean L2/tensor | ~2.6 | ~1.3 |
| Total L2 norm | 34.4 | |

The lora_b weights are **non-zero** (mean_abs ~0.004), confirming training DID modify them from their zero initialization. The adapter is not empty — it actively learned something.

### What It Learned

Looking at the Base+LoRA trajectories, the adapted model produces output like:

```
"you should you you to use the you use to find you use not be be be be be be be
be include the error to use this to find you use not be be not be not be include
the error to use the error?<|im_end|>!"
```

Every step: zero tool calls, pure repetitive text with leaked special tokens (`<|im_end|>`). The adapter shifted the model's output distribution toward degenerate repetition rather than useful tool-calling behavior.

### Why the Adapter Damages the Model

The training data consists of 86 short Q&A pairs with no tool-calling structure. The base model was instruction-tuned for chat with XML tool tags (`<bash>`, `<read>`, `<write>`, `<done>`). Training on short-answer Q&A without tool tags likely:

1. **Shifted attention patterns** away from the XML tool-calling format
2. **Compressed the output distribution** toward short, repetitive patterns that were common in the training data
3. **Interfered with the chat template** — the `<|im_end|>` token leaking into generated text suggests the adapter disrupted the model's understanding of the chat template structure

With `alpha/rank = 32/16 = 2.0`, the LoRA scaling factor amplifies these effects. This is aggressive for a 4-bit quantized model.

## Finding 3: Training Data Quality Is Catastrophically Bad

### Phase 1 Synthesis Output

86 synthetic instruction-response pairs generated by the same 7B model from 25 (mostly failed) trajectories. Sample content:

- "How should you read a file using the read tool?" -> "You should use the command 'read' followed by the path..."
- "How should you find all Python files in a directory?" -> "You should use the `find` command..."
- "What is the project-specific convention for organizing code in Django?" -> generic boilerplate

### Problems

1. **Trivial content** — Every 7B model already knows how to read files and find Python code. There is zero skill signal.
2. **No tool-call structure** — The synthesis prompt asks for Q&A pairs, not agent trajectories with tool calls. The adapter learns to produce prose, not tool invocations.
3. **Self-distillation from failures** — 92% of trajectories failed (only 2/25 resolved). Synthesizing from failed trajectories teaches the model to reproduce failure patterns.
4. **7B synthesis quality** — The synthesis model fails ~30% of trajectories on JSON parsing alone.

### Oracle Data for Comparison

The oracle SFT training data contains 20 gold-standard patches from SWE-Bench in a clean format:
- **User**: "Fix the following Django bug. [file path] [bug report]. Provide your fix as a unified diff patch."
- **Assistant**: Actual correct unified diff

This is dramatically better: real signal, correct answers, and directly relevant to the task. The oracle evaluation (below) tests whether LoRA CAN help with good data.

## Finding 4: The Two Resolved Tasks Have Solution Leakage

### django-12304 (Enum `do_not_call_in_templates`)

The bug report says: *"The easy solution would be to declare `do_not_call_in_templates = True` on the various Choices classes."*

The ground truth patch is a single line: `cls.do_not_call_in_templates = True`. The answer is literally in the problem statement.

### django-13410 (Concrete model with `to_field`)

The bug report includes a **complete `diff --git` patch** showing the exact code change needed. The agent essentially copies the fix from the description.

### No Other Test Task Has This

Among the 23 failed test tasks, none have the fix embedded in the problem statement. The 8% base resolve rate (2/25) is entirely explained by solution leakage in the SWE-Bench problem statements — it does not reflect genuine debugging or coding skill.

## Three Hypotheses for FT=0

### Hypothesis 1: Format Mismatch (Training != Inference)
**Claim:** LoRA training on Q&A prose cannot improve tool-calling agent behavior because the formats are fundamentally different.

**Evidence:**
- Training data: short prose Q&A without any tool tags
- Inference: multi-turn agent loop using `<bash>`, `<read>`, `<write>`, `<done>` XML tags
- The adapter produced degenerate output with leaked `<|im_end|>` tokens
- The effective LoRA delta is a distribution shift AWAY from tool-calling

**Test:** Train on oracle data (proper diff patches in response to bug reports) and evaluate. If the adapter helps with oracle data, format mismatch is confirmed as the primary cause.

### Hypothesis 2: Self-Distillation from Failures Cannot Produce Positive Signal
**Claim:** When the base model resolves <10% of tasks, synthesizing training data from its own (failed) trajectories creates a negative feedback loop.

**Evidence:**
- 92% of training trajectories failed (23/25 tasks unresolved)
- Synthesized content reflects failure patterns, not success patterns
- The 2 resolved tasks are trivial (answer in problem statement) — not representative of general skill
- Even the "successful" synthesis pairs are generic ("how to use read tool") not specific to Django debugging

**Test:** Compare oracle adapter (trained on 20 gold patches) vs Phase 1 adapter (trained on 86 self-synthesized Q&A). If oracle >> Phase 1, self-distillation is confirmed as a dead end for low-capability base models.

### Hypothesis 3: 4-bit Quantized LoRA on 7B Is Below the Skill Threshold
**Claim:** A 4-bit quantized 7B model doesn't have enough representational capacity for LoRA to encode useful behavioral skills. The adapter has too few effective degrees of freedom.

**Evidence:**
- 4-bit quantization limits gradient precision during training
- Rank 16 with alpha=32 (scaling 2.0) is aggressive — may overwhelm the quantized weights
- The adapter size (88 MB, 23M params) is ~1.3% of full model params — reasonable for larger models but potentially insufficient for a 7B model where base capabilities are limited
- All Experiment 6/7 results show no differentiation between formats/budgets

**Test:** Run the same experiment with qwen2.5-coder:14b (once pulled). If 14B shows improvement where 7B doesn't, the capacity hypothesis is supported.

Before any future evaluation, `looper/agent/workspace.py` must be fixed to reset workspace state between conditions:

```python
# Current (buggy):
if workspace.exists() and get_head(workspace) == base_commit:
    return workspace  # Returns dirty workspace!

# Fix:
if workspace.exists() and get_head(workspace) == base_commit:
    subprocess.run(["git", "checkout", "--", "."], cwd=workspace)
    subprocess.run(["git", "clean", "-fd"], cwd=workspace)
    return workspace
```

## Oracle SFT Evaluation Results

**Date:** 2026-03-09 (13:51 – 19:22 PST, ~5.5 hours)

### Setup

- **Training data:** 20 gold SWE-Bench patches (user=bug report, assistant=unified diff)
- **Validation data:** 5 gold patches
- **Training:** MLX LoRA, rank 16, 100 iterations, batch_size=1, max_seq_length=2048
- **Training metrics:** train_loss 1.836 → 0.659, val_loss 1.836 → 2.644 (overfitting on 20 examples)
- **Adapter:** 88 MB, 23M parameters (same architecture as Phase 1)
- **Workspace bug fixed:** `reset_workspace()` called before each condition

### Results

| Condition | Resolved | Rate | Patches Generated |
|-----------|----------|------|-------------------|
| Base (Ollama) | 2/25 | 8.0% | 9/25 |
| Oracle-adapted (MLX) | 0/25 | 0.0% | 0/25 |

**Forward transfer: -0.08** (negative — the adapter made the model strictly worse)

Base resolved the same 2 tasks as Phase 1: django-12304 and django-13410 (both have solution leakage in the problem statement).

### What the Oracle Adapter Does

The adapted model generates **raw diff output** directly instead of using tool calls:

```
Step 0 reasoning: "diff --git a/django/template/templatetags/enum.py
b/django/template/templatetags/enum.py
--- a/django/template/templatetags/enum.py
+++ b/django/template/templatetags/enum.py
@@ -30,6 +30,7 @@ def _get_repr_opts(cls):
 class Enum(models.Model):
     class Meta:
         get_auto_index = False
+    _meta = None
..."
```

The training data taught "bug report → diff patch". The adapter learned this mapping faithfully — but the agent loop expects `<bash>`, `<read>`, `<write>`, `<done>` XML tool calls. The result:

1. **Every step:** model outputs a hallucinated diff (nonexistent files, wrong code)
2. **Zero tool calls parsed:** no `<bash>`, `<read>`, `<write>` tags in output
3. **No workspace interaction:** the agent never reads files, runs commands, or writes patches
4. **All 25 tasks:** hit max_steps (15) with no generated patches
5. **~12 min per task** with MLX (vs ~1.5 min with Ollama) due to in-process inference

### Hypothesis Evaluation

| Hypothesis | Verdict | Evidence |
|-----------|---------|----------|
| H1: Format mismatch | **CONFIRMED** | Oracle adapter produces diffs, agent needs tool calls. Even gold data in wrong format = negative transfer. |
| H2: Self-distillation from failures | **CONFIRMED (partially)** | Phase 1 data was garbage, but even oracle data doesn't help — format mismatch is the dominant failure mode. |
| H3: 4-bit 7B below threshold | **INCONCLUSIVE** | Cannot separate capacity limits from format mismatch with current data. |

### Implications

1. **LoRA CAN learn from training data** — train loss dropped from 1.8 to 0.7, the adapter weights are non-trivial, and the model faithfully reproduces the training format (diffs).
2. **Format alignment is non-negotiable** — training data MUST match the inference task format. For an agent that uses XML tool calls, training data must contain XML tool calls.
3. **The synthesis pipeline is fundamentally broken** — it produces Q&A pairs (Phase 1) or would need to produce multi-turn agent trajectories with tool calls. Current synthesizers cannot do this.
4. **Next step:** Train on successful agent trajectories (with tool calls) from the base condition, not on diffs or Q&A pairs. This requires solving the cold-start problem (need successful trajectories to train, but need training to get successful trajectories).

---

## Conclusions

The Phase 1 FT=0 result and the Oracle SFT FT=-0.08 result are explained by a cascade of failures:

1. **Infrastructure bug** (workspace contamination) invalidated the Phase 1 cross-condition comparison — now fixed
2. **Format mismatch is the dominant failure mode** — both Q&A pairs (Phase 1) and diff patches (Oracle SFT) damage the model because neither matches the agent's XML tool-calling format. The oracle experiment proves this: even with perfect answer content, wrong format = negative transfer.
3. **Data quality matters but is secondary** — Phase 1's self-distilled Q&A from failures was garbage, but even oracle gold data doesn't help when the format is wrong.
4. **The adapter actively damages the model** — in both experiments, the adapted model produces zero tool calls and hits max_steps on every task. LoRA scaling (alpha/rank=2.0) on 4-bit quantized weights amplifies the distribution shift away from tool-calling behavior.

**The core thesis remains untested.** Neither experiment trained on data that matches the inference format. To test whether LoRA can encode useful agent skills, training data must consist of multi-turn agent trajectories with XML tool calls — not Q&A pairs, not standalone diffs. This is the critical gap.
